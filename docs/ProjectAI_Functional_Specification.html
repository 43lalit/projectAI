ProjectAI (FedMDRM) Functional Specification

Repository Basis: ProjectAI Spring Boot application source code and static UI pages.

Document Version: 1.0

Date: 2026-03-03

1. Purpose

This document defines the functional behavior of ProjectAI, a web application for MDRM data ingestion, discovery, reporting analysis, profile exploration, bookmarks, and recent-activity tracking. It specifies user-facing functionality, backend behavior, API contracts, business rules, data persistence, authorization rules, and operational flows as implemented in the current repository.

2. Product Scope

ProjectAI provides:

  • MDRM data ingestion from classpath files and uploaded files, with run-based lineage.
  • Run summary and incremental change analytics across reporting forms.
  • Discovery search (keyword and semantic query pattern) with run-context time travel.
  • Report-centric analysis with KPIs, history, and drilldown.
  • Detailed MDRM profile page with timeline, association mapping, and related MDRM graph.
  • User authentication (sign up, login, logout, password reset) with session + CSRF protection.
  • User bookmarks (groups and items), run-context-aware.
  • User recent item history, run-context-aware with server and local fallback.
  • Onboarding guide and command palette for navigation/productivity.

3. System Context

3.1 Technology Stack

  • Backend: Java 17, Spring Boot 3.3.5, Spring Web, Spring JDBC.
  • Database: PostgreSQL (required for MDRM promotion function).
  • Security components: BCrypt password hashing (spring-security-crypto), session + custom CSRF token.
  • CSV parsing: Apache Commons CSV.
  • Frontend: Static HTML/CSS/JavaScript, Bootstrap, AG Grid, D3 (profile graph), Tailwind utilities on main page.

3.2 Runtime Entry Points

  • GET / forwards to /index.html.
  • Main console page: /index.html.
  • MDRM profile page: /mdrm.html.

6. Functional Requirements

6.1 Authentication and Security

  • FR-AUTH-001: System shall support user registration with first name, last name, email, and password.
  • FR-AUTH-002: Registration email shall be normalized to lowercase and must be unique.
  • FR-AUTH-003: Password length shall be at least 6 characters for register and reset.
  • FR-AUTH-004: System shall support login using email + password and set session user ID on success.
  • FR-AUTH-005: System shall expose /api/auth/me to return authentication status and user details.
  • FR-AUTH-006: System shall support logout by invalidating the session.
  • FR-AUTH-007: All mutating endpoints protected by CSRF shall require header X-CSRF-Token.
  • FR-AUTH-008: System shall expose /api/auth/csrf to issue/retrieve session CSRF token.
  • FR-AUTH-009: Admin-only endpoints shall require authenticated user with role ADMIN.

6.2 MDRM Ingestion and Run Management

  • FR-LOAD-001: Manual full migration shall be triggered via POST /api/mdrm/load (admin + CSRF).
  • FR-LOAD-002: Full migration shall clear MDRM operational tables and process all matching classpath files oldest-first.
  • FR-LOAD-003: Uploaded file ingestion shall be available via POST /api/mdrm/upload (multipart, admin + CSRF).
  • FR-LOAD-004: Uploaded file name shall match MDRM_ddyy.csv.
  • FR-LOAD-005: Uploaded headers shall match current staging header structure when existing structure is present.
  • FR-LOAD-006: CSV parser shall treat line 1 as metadata and line 2 as header row.
  • FR-LOAD-007: Headers shall be sanitized to unique DB-safe snake_case identifiers.
  • FR-LOAD-008: System shall recreate staging table per run based on sanitized headers plus run_id.
  • FR-LOAD-009: Promotion to master shall be executed through PostgreSQL function promote_mdrm_run.
  • FR-LOAD-010: Rows with invalid or blank date fields shall be written to run error table and excluded from master.
  • FR-LOAD-011: System shall derive mdrm_code = mnemonic + item_code.
  • FR-LOAD-012: System shall derive is_active based on end date (year 9999 or future => Y, else N).
  • FR-LOAD-013: System shall compute and persist run summary, incremental summary, file summary, and report status.
  • FR-LOAD-014: For uploaded no-op runs (no incremental changes vs previous), run data shall be rolled back/deleted and request shall fail.

6.3 Discovery, Reporting, and Profile Querying

  • FR-QUERY-001: System shall provide reporting forms list and status list for latest run or selected as-of run.
  • FR-QUERY-002: System shall return tabular rows for a reporting form from the latest eligible run (run_id <= selectedRun when context provided).
  • FR-QUERY-003: System shall provide run history including total/active/inactive/updated and incremental add/modify/delete counts.
  • FR-QUERY-004: System shall provide MDRM code drilldowns by run bucket (TOTAL/ACTIVE/INACTIVE/UPDATED).
  • FR-QUERY-005: System shall provide incremental MDRM code drilldowns (ADDED/MODIFIED/DELETED).
  • FR-QUERY-006: Semantic search shall support keyword extraction with stop-word removal and optional inferred reporting form.
  • FR-QUERY-007: Semantic search shall return up to 500 results with interpreted query metadata.
  • FR-QUERY-008: MDRM profile API shall return consolidated profile data (overview, timeline, field changes, associations, related MDRMs/reports).

6.4 Bookmarks and Recent Activity

  • FR-BM-001: Authenticated users shall manage bookmark groups (list, create, rename, delete).
  • FR-BM-002: Authenticated users shall manage bookmark items (list, add, move group, delete).
  • FR-BM-003: Bookmark uniqueness shall be enforced per user + item_type + item_key + group + run_context.
  • FR-BM-004: Bookmark and group actions shall validate ownership and reject invalid group IDs.
  • FR-REC-001: Authenticated users shall list, upsert, and clear recent items per run context.
  • FR-REC-002: Recent upsert shall be atomic via DB conflict update on user/type/key/run_context.
  • FR-REC-003: Recent list limit shall be clamped to 1..50 server-side.

6.5 Frontend Interaction Requirements

  • FR-UI-001: Main page shall provide four primary views: Discovery, Bookmarks, Load Data, Reports.
  • FR-UI-002: Header run selector shall apply global as-of run context across discovery/reporting/bookmarks/recent workflows.
  • FR-UI-003: Command palette shall support keyboard-driven quick actions and deep-link navigation.
  • FR-UI-004: Guide overlay shall provide step-based onboarding with first-visit behavior.
  • FR-UI-005: UI shall support light/dark themes with persisted preference.
  • FR-UI-006: MDRM profile page shall provide tabs: Overview, Timeline, Report Association, Related MDRM Graph.
  • FR-UI-007: Deep links shall support discovery/report/reporting/profile state sharing with run context.

8. Data Model Specification

8.1 Core MDRM Tables

  • mdrm_run_master: run metadata (run_id, run_datetime, file_name, file/ingest/error counts).
  • mdrm_staging: per-run recreated staging with dynamic CSV columns + run_id.
  • mdrm_master: canonical persisted rows, includes dynamic source columns and derived fields.
  • mdrm_run_error: row-level rejection log with raw record and error description.
  • mdrm_run_summary: per run/reporting_form aggregate counts.
  • mdrm_run_incremental: per run/reporting_form added/modified/deleted counts versus previous run.
  • mdrm_file_summary: per run file-level KPI rollup.
  • mdrm_report_status: latest status snapshot by reporting form.

8.2 User and Personalization Tables

  • app_user: identity, role, hashed password, profile fields.
  • favorite_group: user-defined bookmark groups.
  • bookmark_item: bookmarked entities with optional group and optional run context.
  • recent_item: recent entities with run-context dimension and last_accessed ordering.
  • report_metadata: cached report full name and description fetched from Federal Reserve pages.

8.3 Key Indexing/Constraints

  • Unique email index on app_user(email).
  • Unique bookmark group name per user.
  • Recent uniqueness: (user_id, item_type, item_key, run_context_id).
  • Summary/incremental primary keys by (run_id, reporting_form).

9. Business Rules and Calculations

  • BR-001: Run ID is epoch milliseconds and must be unique; collisions are incrementally retried.
  • BR-002: reporting_form and MDRM codes are required for summary participation.
  • BR-003: Run bucket classification uses code-level flags:
    • ACTIVE: has active only.
    • INACTIVE: has inactive only.
    • UPDATED: has both active and inactive.
  • BR-004: Incremental MODIFIED compares row count, active/inactive flags, min start date, max end date.
  • BR-005: Semantic search removes common stop words and attempts reporting-form inference from known forms.
  • BR-006: Item type code mapping (J, D, F, R, S, E, P) is normalized to business labels in outputs.
  • BR-007: Run context semantics are as-of: data selected with run_id <= selectedRunId where implemented.

10. Screen-Level Functional Behavior

10.1 Main Console (index.html)

  • Persistent header with brand navigation, menu tabs, command palette, quick tour, theme toggle, run context selector, auth menu.
  • Discovery view:
    • Search box and filters.
    • Recent searches and saved searches.
    • AG Grid result table (bookmark action integration).
    • Recent items strip with bookmark toggles and reopen behavior.
  • Bookmarks view:
    • Group management and item listing.
    • Create/rename/delete groups; move/remove items.
  • Load Data view:
    • Admin-only load/upload actions.
    • File run metrics and incremental details.
  • Reports view:
    • Reporting form list/status filters.
    • Run history KPIs and compare controls.
    • Drilldown panels for active and net change buckets.
    • Report metadata display and admin refresh action.

10.2 MDRM Profile Page (mdrm.html)

  • Header with same auth + theme controls as main console.
  • MDRM-specific breadcrumb and share-link copy action.
  • Tabbed panels:
    • Overview: high-level profile summary + first vs latest field deltas.
    • Timeline: run events, statuses, per-run field changes.
    • Report Association: report/schedule/line mapping.
    • Related MDRM Graph: D3 concentric graph with detail panel and active/inactive filtering behavior.
  • Profile load supports optional run context from query parameters.
  • Recent activity is recorded locally and server-side when authenticated.

11. External Integration

11.1 Federal Reserve Reporting Forms

  • Report metadata service fetches HTML from https://www.federalreserve.gov/apps/reportingforms/Report/Index/{REPORT}.
  • Service extracts full report name and description using multiple regex fallback strategies and caches results in DB.
  • Fetch failures return HTTP 502 semantics via Spring ResponseStatusException.

12. Scheduling and Operations

  • Scheduled job @Scheduled(cron = ${mdrm.cron}) runs full migration load periodically.
  • Default cron in configuration: 0 0 1 * * * (daily 01:00 server time).
  • Startup schema initialization ensures all required tables/indexes/functions and performs backfills for missing summary artifacts.

13. Error Handling and Validation

  • Validation and business-rule failures raise typed HTTP errors via ResponseStatusException.
  • Common statuses:
    • 400: invalid input (missing fields, unsupported bucket/change type, invalid group ID).
    • 401: authentication required / invalid credentials.
    • 403: invalid CSRF token / admin required.
    • 404: not found (bookmark/user cases).
    • 409: duplicates (email, group, bookmark in same scope).
    • 500/502: backend and upstream fetch issues.

14. Non-Functional Requirements (Implemented/Implied)

  • NFR-001: Performance support via summary tables and indexes for run/reporting-form queries.
  • NFR-002: Session cookie hardening configured (httpOnly, sameSite strict, configurable secure flag).
  • NFR-003: Passwords stored as bcrypt hashes.
  • NFR-004: Frontend responsive behavior for desktop and smaller viewports.
  • NFR-005: Data consistency for recent upserts through DB-level upsert constraints.

16. Assumptions and Constraints

  • System assumes PostgreSQL for MDRM promotion function; non-PostgreSQL execution is rejected for promotion path.
  • MDRM source content must provide expected date and key columns (start_date, end_date, mnemonic, item_code, reporting_form).
  • Role elevation to ADMIN is email-based for the primary admin email constant.
  • Frontend is static-file served and tightly coupled to current API response shapes.

17. Out of Scope (Current Implementation)

  • OAuth/SSO or enterprise identity providers.
  • Fine-grained RBAC beyond USER vs ADMIN.
  • Distributed/session-store clustering behavior details.
  • Formal API versioning and backward-compatibility policy.
  • Automated migration tooling (Flyway/Liquibase) for DB schema governance.

  • AC-001: Admin can execute full and uploaded MDRM loads with resulting summaries and history available.
  • AC-002: Discovery and report workflows return data scoped to selected run context.
  • AC-003: Profile page provides timeline, association, and related graph from profile API.
  • AC-004: Authenticated users can persist bookmarks and recent items by run context.
